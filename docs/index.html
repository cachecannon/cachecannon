valkey-lab

A deep-dive benchmarking tool for Valkey and Redis. Go beyond quick throughput checks. Mix workloads, control rate, watch tail latency per second, and find your server's saturation point.

Get Started GitHub
Get Started

Install and explore

One cargo install, then pick a scenario. Each terminal below is a real command you can run.

install
# build from source (requires Linux, io_uring) cargo install --path . --bin valkey-lab # or from the APT / YUM repo apt install cachecannon
quick start: just run it
# zero-config defaults: localhost:6379, 60s, 80:20 GET:SET valkey-lab # custom host, 16 connections, pipeline depth 32 valkey-lab -h 10.0.0.1 -c 16 -P 32
realistic workloads
# Zipfian hot-key distribution, write-heavy valkey-lab -r 20:80 --distribution zipf -s 512 # prefill the cache, then benchmark GETs only valkey-lab --prefill -r 100:0 # measure latency at exactly 500K req/s valkey-lab --rate-limit 500000 -c 16 -P 32
saturation search + export
# find the max rate where p999 stays under 1ms valkey-lab saturate --slo-p999 1ms -c 16 -P 32 # save full time-series to Parquet, then view in browser valkey-lab --parquet results.parquet -d 5m valkey-lab view results.parquet

Flags follow valkey-benchmark conventions: -h, -p, -c, -P, -d, -t. Full reference in VALKEY-LAB.md.

Output

Designed to read, not to parse

Benchmark output is something you stare at. valkey-lab uses a deliberate color hierarchy so the numbers you care about stand out and the rest stays quiet. req/s and p999 are bold white. Everything else is dimmed. Errors turn red the instant they appear. The header reprints every 25 rows so you never lose context during a long run.

This is not cosmetic. When you're watching a 5-minute saturation search or debugging a latency spike in production, the difference between scanning a wall of monochrome numbers and seeing the one column that matters bolded is the difference between catching the problem and scrolling past it.

Bold what matters

req/s and p999 are bold white. Time, p50, p90, p99, p9999, max, hit% are all dimmed. Your eye goes straight to throughput and the latency percentile that usually trips SLOs.

Color-coded signals

Cyan for config labels. Green for PASS. Red for FAIL and errors. Yellow for throughput warnings. Dim for separators and low-signal columns. Every color has a reason.

Per-second streaming

One row per second with full p50 through p9999 and max. Spikes that disappear in aggregate percentiles show up as individual rows in the time series.

Why a lab?

Go deeper than a quick benchmark

valkey-benchmark is good for quick throughput checks. valkey-lab builds on that foundation with controls for the questions that come next: what happens under mixed traffic, at a controlled rate, with realistic key distributions, over time?

Mixed GET/SET

Run GETs and SETs concurrently at any ratio (-r 80:20) to see how your server handles real contention patterns.

Rate limiting

Pin the load to a specific request rate (--rate-limit 500000) and measure latency under controlled conditions.

Prefill & warmup

--prefill populates the keyspace before benchmarking. --warmup discards early samples so you measure steady state.

Per-second tail latency

p50 through p9999 and max, every second. Spikes that hide in aggregate percentiles show up in the time series.

Key distribution

--distribution zipf models a realistic hot-key pattern. Uniform is the default; Zipfian reflects production skew.

Exportable results

--parquet results.parquet saves full time-series data. valkey-lab view opens an interactive web dashboard.

Saturation Search

Find max SLO-compliant throughput

Ramps request rate geometrically until your latency SLO breaks. Classifies failures as throughput-limited (server can't keep up) or latency-exceeded (server keeps up but tail latency blows). 

valkey-lab saturate --slo-p999 1ms -c 16 -P 32
